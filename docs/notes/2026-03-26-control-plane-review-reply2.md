> 复核目的: 对第一轮 reply 进行技术与实施层面的再校准，识别可直接采纳项、需修正项与不建议采纳项

## 0. 复核结论摘要

第一轮 reply 整体质量是可用的，尤其在以下方面判断准确：

1. API 契约必须补齐，且应从方向性描述变成可测试基线。
2. 审计能力不能只停留在 PRD 承诺，必须进入数据模型与验收项。
3. 当前仓库基线与目标态架构之间存在明显脱节，进度文档需要补齐基线差距说明。
4. HookRunner 与 EventBus 不应混为同一阶段能力。
5. 实施计划中存在 Create/Modify 描述失真与已知技术债遗漏。

但第一轮 reply 有 4 个问题不适合直接采纳，否则会误导后续实施顺序：

1. 过度下调安全问题优先级，并提出 shared secret 过渡方案。
2. 过度否定持久化访问边界抽象的必要性。
3. 低估容器化与多进程部署形态对中期验证闭环的影响。
4. 将 Phase DoD 简化得过轻，不足以形成稳定评审基线。

**最终判断：**

- 第一轮 reply 可以作为第二轮修订输入材料。
- 不应直接作为最终修订执行基线。
- 建议在吸收本轮复核意见后，再形成最终版修订清单。

## 1. 可直接采纳的判断

### 1.1 API 契约需要独立补齐

第一轮 reply 对此判断准确。

当前设计文档虽然已经列出了路由前缀和最小 job 返回示例，但仍不足以支撑后续实现与测试。尤其是以下内容仍缺失：

1. 请求体结构
2. 成功/失败响应 envelope
3. 状态码规范
4. 幂等语义
5. Job 状态机
6. Agent 端接口的输入输出约束

这部分应形成独立契约文档，而不是继续堆在 design 文档正文中。

**结论：采纳。**

---

### 1.2 审计能力必须从承诺转为设计对象

第一轮 reply 关于 AuditEvent 的判断成立。

PRD 已将审计列入核心价值，但当前设计 ER 中没有相应实体，验收标准中也没有“关键操作可查询审计记录”的落地描述。这会让审计从产品承诺退化为“未来可能补”的软目标。

**结论：采纳。**

建议保持以下最小字段：

- actor
- action
- resource_type
- resource_id
- result
- correlation_id
- source_node_id
- created_at

---

### 1.3 当前仓库基线需要显式建模

第一轮 reply 对当前代码基线的识别是准确的：

1. `config.py` 已有 run_mode/control_plane/node_identity/hooks 相关配置结构。
2. `models/` 下的领域模型文件已存在。
3. `services/cert_service.py` 已存在并承担 issue/renew/check 主要编排。
4. `Dockerfile`、`docker-compose.yml`、`pyproject.toml` 仍然是单入口、单服务现状。

如果 progress 文档继续只写目标任务，不写仓库现状，那么后续每个 Task 都会继续出现“文档以为未开始，代码其实半完成”的错位。

**结论：采纳。**

---

### 1.4 HookRunner 与 EventBus 必须解耦表达

第一轮 reply 关于 H3 的分析合理。

当前设计图把 HookRunner 画在 EventBus 下游，但 progress 文档又把 HookRunner 放在 Phase 1、EventBus 放在 Phase 5。这个表达会直接误导实现顺序。

本地同步 hook 与后期领域事件/远端 webhook 属于两类能力：

1. HookRunner: 同步 shell 执行器，属于本地交付闭环。
2. EventBus: 事件发布机制，属于后续系统化事件流。
3. Webhook: 事件外发消费者，依赖 EventBus，但不应反向影响 HookRunner 的前置实现。

**结论：采纳。**

---

### 1.5 `_entry_domains` 重复与 Task 描述失真属于有效补充发现

第一轮 reply 新增的 E1 与 E2 都成立：

1. `_entry_domains` 在 CLI 与 CertService 中重复实现，属于明确的 DRY 违背。
2. 实施计划中多个 Task 写的是 Create，但目标文件已存在，容易误导执行者错误重建。

**结论：采纳。**

## 2. 需修正后采纳的判断

### 2.1 安全问题不应被下调到可以用 shared secret 过渡

第一轮 reply 对“安全验收矩阵不必在当前阶段全部写完”的判断有一定合理性，但其给出的过渡方案不建议采纳。

问题在于：

1. Agent 模式一旦进入真实 poll/bundle/result 链路，安全边界就不是“后期再补”的附属能力，而是协议基线的一部分。
2. 如果 MVP 阶段先用 shared secret，后续切到 Ed25519/X25519 往往不是替换实现，而是要重做消息结构、注册流程、兼容路径与节点升级策略。
3. 当前设计文档已经把 Ed25519 与 X25519/AES-GCM 作为正式设计决策，继续引入弱化版过渡协议会制造双轨实现。

**修正后结论：部分采纳。**

可接受的修订方向应是：

1. 不要求当前就写完整安全验收矩阵。
2. 但必须在进入 Agent 实现前固定最小安全基线：
   - 节点注册与信任建立流程
   - 签名字段与验签边界
   - bundle 下载授权边界
   - 失败事件的审计记录要求
3. 不采用 shared secret 作为正式过渡主方案。

---

### 2.2 不需要重型 Repository Pattern，但需要轻量持久化边界

第一轮 reply 反对“额外 Repository 抽象”的出发点可以理解，但结论过于绝对。

需要区分两件事：

1. **不需要**引入完整 DDD 风格、层层包装的重型 Repository Pattern。
2. **需要**定义清晰的数据访问边界，避免 Service 层未来直接散落 ORM Session 与查询逻辑。

原因：

1. 当前目标已不只是单一 CLI，而是包含 server、worker、scheduler、agent 的多运行实例协作。
2. 一旦引入 Job、Node、Webhook、Audit 这些持久化对象，服务层若直接耦合 ORM 细节，测试难度与演进成本都会上升。
3. Alembic 解决的是 schema migration，不解决服务层如何隔离持久化细节的问题。

**修正后结论：部分采纳。**

建议采用折中方案：

- 不在文档里坚持“必须上完整 Repository Pattern”。
- 但在设计上保留轻量接口或模块级边界，例如 JobStore、AuditStore、NodeStore。

---

### 2.3 Phase DoD 可以轻量，但不能只有一条通用模板

第一轮 reply 反对为每个 Phase 增加过重流程，这个方向本身是对的；但给出的 DoD 简化方案仍然偏轻。

统一模板如：

`验证命令通过 + 新增测试全绿 + 覆盖率 >= 80% + 无 Critical lint issue`

只能解决“形式一致”，不能解决“阶段差异”。

因为不同 Phase 的退出标准本质不同：

1. DB 阶段需要 migration 证据。
2. API 阶段需要契约与错误语义证据。
3. Security 阶段需要握手、验签、失败分支样例。
4. Delivery/Agent 阶段需要交付与回执闭环样例。

**修正后结论：部分采纳。**

建议改为：

1. 每个 Phase 保留 1 条通用 DoD。
2. 再增加 2-4 条 phase-specific exit checks。

这样既不会过重，也能保证评审可操作性。

## 3. 不建议直接采纳的判断

### 3.1 “Phase 0 前不需要安全基础设施”不建议采纳

这不是说 Phase 0 要完整实现全部安全模块，而是说：

如果 Phase 0 已经包含数据库基础设施、依赖引入、模型补强，那么安全最小基线就不能完全留到后面再思考。

否则会出现：

1. Job/Node/Audit 数据模型先做一版。
2. Agent poll/bundle/result 先跑一版。
3. 到 Phase 4 才发现消息结构、节点注册与下载授权模型需要重构。

这类返工不是“正常迭代”，而是架构前置边界没有先钉住。

**结论：不建议采纳该表述。**

---

### 3.2 “修完第 5 节即可作为 Phase 0~1 实施基线”表述过于乐观

第一轮 reply 最后的判断偏积极。

更稳妥的说法应是：

1. 修完第 5 节后，可以作为 **Phase 0 文档修订与核心重构基线**。
2. 若要进入包含 DB 和 Agent 协议边界的实现，还需额外补齐：
   - 最小持久化访问边界说明
   - 最小安全基线说明

**结论：不建议原样采纳该结论。**

---

## 4. 第一轮 reply 的遗漏项

### 4.1 容器化与多进程编排的重要性被低估

第一轮 reply 已识别当前 Dockerfile/Compose 是单入口、单服务，但仍低估了这个问题在中期实施中的影响。

当前事实：

1. Dockerfile 只有 `uv run certman` 单入口。
2. docker-compose.yml 只有单个 certman 服务。
3. 设计文档已经明确存在 `certman-server`、`worker`、`scheduler`、`certman-agent` 多实例形态。

如果把这部分完全压到最后，Phase 3 之后的大量验证都只能在“开发者本机单进程假设”下完成，无法尽早验证部署拓扑是否站得住。

**补充建议：**

至少应将以下内容前移到 Phase 3 左右完成：

1. 多入口脚本注册
2. 基础镜像支持多命令启动
3. server/worker 的最小 Compose 骨架

不要求立即完整生产部署，但不能一直停留在单入口镜像。

---

### 4.2 数据迁移之外，备份恢复与回滚策略仍未进入基线

第一轮 reply 提到了 Alembic，但没有充分展开数据回滚与运维恢复问题。

控制平面的关键数据不是只有 schema：

1. Job 状态
2. 节点身份信息
3. 审计记录
4. 证书运行态目录与凭据目录

仅说明 “SQLite + Alembic migration” 并不能覆盖：

1. migration 失败如何回滚
2. data/run 下关键状态如何备份
3. agent/server 版本不一致时如何处理历史任务数据

**补充建议：**

在后续 runbook 或 ADR 中最少增加：

1. 备份恢复范围
2. migration 前检查点
3. 回滚约束与人工干预点

---

## 5. 调整后的修订建议

基于第一轮 reply 与本轮复核，建议最终修订拆成三层：

### 第一层：立即修订

1. PRD 状态从 `Draft Accepted` 改为单一状态。
2. PRD 验收语句改为可测试表达。
3. progress 文档增加当前仓库基线章节。
4. progress 文档为每个 Phase 增加通用 DoD 与专项退出项。
5. 实施计划校准 Create/Modify 描述。
6. 删除 Task 8 中“先用内存 job store”表述。
7. 设计图中修正 HookRunner 与 EventBus 关系。
8. 设计 ER 中加入 AuditEvent。

### 第二层：进入 DB / Agent 开发前补齐

1. 新增 API 契约文档。
2. 明确最小安全基线。
3. 明确最小持久化访问边界。
4. 增加 server 模式配置要求。
5. 增加 Agent / Server 版本兼容说明。

### 第三层：中期实现前移项

1. 多入口脚本与镜像支持。
2. server/worker 最小 Compose 骨架。
3. Alembic migration 基线。
4. 备份恢复与回滚最小策略。

## 6. 最终结论

对第一轮 reply 的综合判断如下：

| 项目 | 结论 |
|---|---|
| 可直接吸收的主体内容 | 较多 |
| 需要校正后再吸收的内容 | 明显存在 |
| 不宜原样执行的关键点 | 2 项 |
| 是否可直接作为最终执行基线 | 否 |

**最终建议：**

将第一轮 reply 视为“高质量但尚未收口的中间稿”。

下一步应以本文件为准，产出一版最终修订清单，作为：

1. 文档修订执行基线
2. 后续实现计划的真实入口条件

在未补齐“最小安全基线”和“最小持久化访问边界”之前，不建议直接进入 Agent/Server 主体实现。
