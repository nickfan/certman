# CertMan Phase 5 开发计划（Node Registry & Agent Mode）

## 1. 计划概览

- 计划名称：Phase 5 Node Registry & Agent Mode
- 目标周期：3 周（建议 15 个工作日）
- 优先级：高
- 交付原则：先最小可交付，再增强可靠性与部署联动
- 当前基线：
  - 已有 node-agent 轮询与结果回执接口（/api/v1/node-agent/poll, /api/v1/node-agent/result）
  - 已有 NodeORM、NodeNonceORM 及 nonce 重放防护
  - POST /api/v1/nodes/register 仍处于待实装阶段

## 2. 范围定义

### 2.1 In Scope

- Node Registry 的最小闭环：节点注册、身份校验、状态管理（active/pending/disabled）
- Agent Mode 从骨架到可执行闭环：注册 -> 轮询 -> 执行 -> 回执
- 基于 pytest 的 API、服务层、端到端验证
- compose 与 k8s（kind）环境联动验证与文档落盘

### 2.2 Out of Scope

- OIDC 用户认证与细粒度 RBAC
- mTLS 全量体系
- 生产级多租户授权模型
- 跨云对象存储分发适配器

## 3. 技术决策与库选型（2024-2025 活跃优先）

### 3.1 采用方案

- FastAPI（已在项目中）：API 层
- SQLAlchemy + Alembic（已在项目中）：持久化与迁移
- cryptography（已在项目中）：公钥解析与签名能力
- pytest（已在项目中）：测试基线
- httpx（已在项目中）：Agent 与 Server 通信

### 3.2 关键实现策略

- 注册 token 校验：优先使用一次性 token（数据库持久化 + TTL + 单次消费）
- 节点公钥校验：PEM 解析 + 算法约束（Ed25519）+ 指纹记录
- 幂等策略：
  - 注册幂等键：node_id + public_key_fingerprint
  - DB 唯一约束 + upsert/冲突回读
- 重放防护：沿用 node_nonce 机制并扩展到注册相关安全事件

### 3.3 备选方案与取舍

- 备选 A：Redis 作为幂等缓存
  - 优点：高性能
  - 缺点：引入新运行时依赖，不符合当前最小化原则
- 备选 B：JWT 自包含注册 token
  - 优点：无状态
  - 缺点：吊销与单次消费复杂度高
- 结论：Phase 5 采用数据库驱动的一次性 token，更贴合当前架构与运维复杂度

## 里程碑 M1：POST /api/v1/nodes/register 最小可交付（必须先行）

### 目标

- 交付可用的节点注册接口，完成 token 校验、节点公钥校验、注册幂等

### 任务清单

- 定义注册请求/响应 schema（node_id, node_type, register_token, public_key）
- 新增注册 token 存储模型与迁移（含 expires_at、used_at、status）
- 实装 POST /api/v1/nodes/register：
  - 校验 token 存在、未过期、未使用
  - 校验公钥 PEM 可解析且算法符合约束
  - 幂等处理：重复注册返回同一节点记录，不重复写脏数据
- 记录审计事件（注册成功/失败、失败原因）
- 提供最小管理命令或脚本：生成一次性注册 token（开发/测试使用）

### pytest 测试点（至少 2 项）

- 测试点 1：有效 token + 合法公钥 -> 201/200 且 node 状态为 active（或 pending，按设计）
- 测试点 2：重复请求（同 node_id + 同公钥）-> 幂等返回，不新增重复记录
- 测试点 3：token 失效（过期/已使用）-> 401/422 且错误码明确
- 测试点 4：非法公钥（格式错误/算法不符）-> 422

### 验收标准

- API 合约稳定：错误码覆盖 token 无效、公钥非法、语义冲突
- 注册幂等在数据库层可证明（唯一约束 + 回读一致）
- 审计日志可追踪每次注册尝试
- 相关 pytest 全通过

### 风险与缓解

- 风险：token 被重放
  - 缓解：单次消费 + used_at 原子更新 + 短 TTL
- 风险：公钥算法混用导致后续验签失败
  - 缓解：注册阶段强制算法白名单，落库存指纹和算法类型

### 文档与部署联动

- 更新 docs/zh-CN 与 docs/en 的 Node 注册章节（请求示例、错误码、幂等语义）
- compose：新增开发辅助命令（生成 register token）
- k8s：在 ConfigMap/Secret 模板中补充 node 注册参数示例

## 里程碑 M2：Agent 首次启动注册闭环

### 目标

- Agent 启动时可自动完成首次注册并持久化本地身份状态，随后进入轮询

### 任务清单

- 扩展 node_identity 配置（register_token_path 或环境变量注入）
- NodePoller 增加 bootstrap 流程：
  - 未注册时调用 /api/v1/nodes/register
  - 注册成功后写入本地状态标记（避免重复首注）
- 失败回退策略：注册失败时不进入执行态，仅保留可观测日志
- 明确节点状态迁移：pending -> active 的策略（当前可先简化为注册即 active）

### pytest 测试点（至少 2 项）

- 测试点 1：Agent 首启自动注册成功后，poll 正常返回 assignments
- 测试点 2：注册失败（token 无效）时，Agent 不执行任务且退出码/日志符合预期
- 测试点 3：已注册节点重启，不重复触发注册（验证幂等短路）

### 验收标准

- Agent 在 once 模式下可观察到 register -> poll 连续行为
- 注册失败与轮询失败日志可区分
- 现有 test_agent_mode 不回归，新增 bootstrap 用例通过

### 风险与缓解

- 风险：首次注册与轮询并发导致状态不一致
  - 缓解：Agent 启动阶段强制串行（先注册后轮询）
- 风险：本地状态文件损坏
  - 缓解：状态文件采用原子写入 + 读失败回退“未注册”路径

### 文档与部署联动

- compose：增加 certman-agent 服务示例（接入 control_plane）
- k8s：增加 certman-agent Deployment 模板（挂载 node 私钥与 token）
- 文档补充 Agent 首启时序图

## 里程碑 M3：Node Registry 与任务分配授权联动

### 目标

- 基于 Node Registry 状态控制任务分配与结果回执授权，形成最小安全边界

### 任务清单

- poll/result 路由统一接入节点状态校验（active 才允许）
- 结果回执增加 job-node 绑定校验强化
- 更新 last_seen 与心跳时间窗口策略
- 补充节点禁用流：disabled 节点拒绝 poll/result

### pytest 测试点（至少 2 项）

- 测试点 1：disabled 节点请求 poll/result -> 401/403
- 测试点 2：node-a 回执 node-b 的 job -> 401
- 测试点 3：active 节点成功 poll 后 last_seen 更新

### 验收标准

- 节点状态与任务授权逻辑一致
- 非法节点无法消费或回执任务
- 安全错误码稳定，便于告警与排障

### 风险与缓解

- 风险：误禁用导致任务堆积
  - 缓解：提供节点状态查询接口与快速恢复操作
- 风险：last_seen 更新频率过高导致 DB 压力
  - 缓解：节流更新策略（例如 30-60 秒窗口）

### 文档与部署联动

- 文档新增 Node 生命周期与状态机说明
- compose/k8s 增加节点禁用场景演练步骤

## 里程碑 M4：Agent 执行闭环增强（下载/执行/回执）

### 目标

- 将 Agent 从“仅轮询骨架”提升为“可执行交付任务并可靠回执”的最小生产闭环

### 任务清单

- 定义 bundle 拉取与本地执行的统一接口（先 filesystem delivery）
- 将 NodeExecutor 接入 Agent 主流程：
  - 接收 assignments
  - 拉取 bundle
  - 执行 hooks
  - 上报 result
- 失败重试策略（轻量版）：
  - 网络失败重试（有限次数）
  - 业务失败不盲目重试，回执 failed

### pytest 测试点（至少 2 项）

- 测试点 1：assignment -> executor 成功写文件并上报 completed
- 测试点 2：hook 失败 -> 上报 failed 且错误信息完整
- 测试点 3：回执签名校验失败 -> 服务端拒绝

### 验收标准

- Agent once 模式可完成单任务端到端
- 成功与失败路径均可在 job 状态中体现
- 本地交付文件与 hook 执行结果可追踪

### 风险与缓解

- 风险：重复执行同一 assignment
  - 缓解：本地短期去重缓存 + 服务端 job 状态幂等
- 风险：bundle 损坏导致执行异常
  - 缓解：下载后 hash 校验，失败立即回执并标记错误码

### 文档与部署联动

- 补充 Agent 执行链路运维手册（故障排查、日志定位）
- compose：加入 server+worker+agent 三服务联调脚本
- k8s：补充最小 e2e 清单（包含 agent）与端口/Service 说明
