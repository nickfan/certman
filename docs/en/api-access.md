# API & AI Access

This document describes the current machine-facing access surface for CertMan.

## 1. Control-plane documentation endpoints

When `certman-server` is running on `http://127.0.0.1:8000`, the live API docs are:

- Swagger UI: `http://127.0.0.1:8000/docs`
- ReDoc: `http://127.0.0.1:8000/redoc`
- OpenAPI JSON: `http://127.0.0.1:8000/openapi.json`

## 2. Official machine-facing interfaces

Current supported interfaces:

- HTTP REST API exposed by `certman-server`
- OpenAPI schema generated by FastAPI
- `certmanctl` as an operator-friendly CLI wrapper over REST
- `certman-mcp` stdio MCP server wrapping control-plane APIs

## 2.1 MCP startup example

```bash
uv run certman-mcp --endpoint http://127.0.0.1:8000
```

The MCP server uses stdio transport and exposes tools for health, certificate jobs, jobs query/wait, and webhook CRUD.

Notes:

1. `cert_create` and `cert_renew` are asynchronous: they return `job_id`; use `job_wait` to wait for terminal state.
2. Current event topics are job-level (`job.queued`, `job.completed`, `job.failed`). Certificate-level events are planned for addon/plugin integration.

## 3. `certmanctl` to REST mapping

| `certmanctl` | REST endpoint |
| --- | --- |
| `health` | `GET /health` |
| `cert create --entry-name <name>` | `POST /api/v1/certificates` |
| `cert list` | `GET /api/v1/certificates` |
| `cert get --entry-name <name>` | `GET /api/v1/certificates/{entry_name}` |
| `cert renew --entry-name <name>` | `POST /api/v1/certificates/{entry_name}/renew` |
| `job get --job-id <id>` | `GET /api/v1/jobs/{job_id}` |
| `job list ...` | `GET /api/v1/jobs?...` |
| `job wait --job-id <id>` | polling `GET /api/v1/jobs/{job_id}` until terminal state |
| `webhook create` | `POST /api/v1/webhooks` |
| `webhook list` | `GET /api/v1/webhooks` |
| `webhook get --id <id>` | `GET /api/v1/webhooks/{id}` |
| `webhook update --id <id>` | `PUT /api/v1/webhooks/{id}` |
| `webhook delete --id <id>` | `DELETE /api/v1/webhooks/{id}` |

## 4. AI integration guidance

Recommended order for AI tooling:

1. Read `/openapi.json`
2. Use the request/response examples from OpenAPI
3. Call REST endpoints directly or shell out to `certmanctl`

Use `certmanctl` when you want stable operator UX and predictable exit codes.
Use REST + OpenAPI when you want typed client generation or direct tool integration.
